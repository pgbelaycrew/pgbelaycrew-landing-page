01 · Install guide

Standing pgBelayCrew up on real infrastructure

For operators deploying on real infrastructure. The install paths differ per component:

  • Console — one deployment per organization. Container image (Docker, Kubernetes/Helm).
  • Agent — one per PostgreSQL host. Native OS package (.deb / .rpm / .apk / .pkg.tar.zst) or container.
  • UI — separate container, served behind the same origin as the console via your reverse proxy.

If you only want to evaluate the product, start with the validation harness below — it deploys all three components on one host with a single docker compose up. Once you've seen it work end-to-end, come back for the per-component install.

Where the agent runs

The agent runs on the PostgreSQL host, not in your Kubernetes cluster. It needs filesystem reach to PGDATA, the WAL archive, and the Postgres Unix socket — that's a host concern, not a pod concern. The console and UI are happy in Kubernetes; the agent is not.

Quick start — validation harness

The fastest way to see pgBelayCrew running. It spins up two real PostgreSQL clusters (single-node and primary + standby), real MinIO for backup storage, real Dex for OIDC, and three real agents — same code paths as production.

Requirements: Docker ≥ 24, Docker Compose v2, ~6 GB RAM available to Docker.

validation
cd validation
./scripts/bootstrap.sh        # generates dev PKI + .env
docker compose up -d          # PG + MinIO + Dex + console + UI + agents
./scripts/enroll-agents.sh    # mints enrollment tokens, attaches the 3 agents
./scripts/smoke.sh            # runs the smoke scenarios

Open http://localhost and sign in via Dex with admin / admin. To wipe everything: ./scripts/reset.sh.

Workstation only

The harness ships dev-only secrets in plain text and binds Dex/MinIO consoles on loopback. It is safe on a workstation, not safe on a shared host. Never expose any of its ports beyond 127.0.0.1.

1 · Console install

The console is the central control plane. One deployment serves all clusters and all human operators.

What you need first

  • A PostgreSQL 14+ instance for the console's own state DB (separate from the clusters you manage).
  • An OIDC provider (Keycloak, Okta, Azure AD, Dex, …) — there is no local user DB. The console authenticates humans only via OIDC.
  • An S3-compatible storage backend for backups (AWS S3, Scaleway, OVHcloud, MinIO, R2…). Configured per cluster, in the UI, not at install time.
  • A reverse proxy / ingress that terminates TLS and routes the same origin to the UI container (port 3000) and the console's /api/, /api/v1/healthz, /pki/ (port 8080).
  • A PKI for the agent ↔ console mTLS plane. The console CA signs agent certs at enrollment time.
  • NTP-synchronised clocks on every host to within ±5 minutes — mTLS and the audit hash chain are unforgiving here. chrony recommended.
  • A secrets store for PGBC_SESSION_SECRET and the OIDC client secret; the unseal password lives outside any of these.

Option A — Container (single host, Docker Compose)

The minimal production-ish topology runs three containers (UI, console, console-state-Postgres) plus your reverse proxy. A ready-to-adapt compose file ships at docker-compose.prod.yml — it wires the four services together (UI, console, console-state-Postgres, HAProxy), pulls every image from ghcr.io/pgbelaycrew/pgbc-*:stable, terminates TLS at HAProxy with hardened cipher suites and HSTS, and routes /api, /pki to the console while everything else goes to the UI on the same origin.

bash
# Fetch just what you need — no clone required.
mkdir -p pgbelaycrew/{certs,haproxy} && cd pgbelaycrew
curl -fsSLO https://raw.githubusercontent.com/<owner>/pgbelaycrew/<tag>/docs/docker-compose.prod.yml
curl -fsSL  https://raw.githubusercontent.com/<owner>/pgbelaycrew/<tag>/docs/haproxy/haproxy.cfg \
    -o haproxy/haproxy.cfg

# pgbelaycrew.env: mode 0600, see header of docker-compose.prod.yml
# certs/pgbelaycrew.pem: full chain + private key, mode 0600
docker compose -f docker-compose.prod.yml --env-file pgbelaycrew.env up -d

Prefer assembling your own compose / k8s manifests? Pull the images and compose the console up with the mandatory production env vars:

docker-compose.yml
services:
  console:
    image: ghcr.io/pgbelaycrew/console:<tag>
    environment:
      PGBC_ENV: production
      DATABASE_URL: postgres://pgbc:***@console-pg:5432/pgbelaycrew?sslmode=require
      PGBC_SESSION_SECRET: ${PGBC_SESSION_SECRET}     # ≥ 32 random bytes
      # No PGBC_MASTER_KEY — generated on first boot, wrapped under
      # an operator-supplied unseal password (see First-boot below).
      PGBC_OIDC_ISSUER:        https://idp.example.com
      PGBC_OIDC_CLIENT_ID:     pgbelaycrew-console
      PGBC_OIDC_CLIENT_SECRET: ${OIDC_CLIENT_SECRET}
      PGBC_OIDC_REDIRECT_URL:  https://pgbelaycrew.example.com/api/v1/auth/callback
      PGBC_UI_BASE_URL:        https://pgbelaycrew.example.com
      PGBC_BOOTSTRAP_ADMIN_EMAIL: ops-lead@example.com  # first verified login → admin
    ports:
      - "127.0.0.1:8080:8080"   # HTTP (proxy-fronted)
      - "9090:9090"             # gRPC, exposed to agents over the network
      - "127.0.0.1:9091:9091"   # Prometheus metrics

Database migrations are applied automatically. The console embeds its schema (goose) and queue (river) migrations and runs them in-process at startup, before it serves any traffic — migrations are idempotent and serialized across replicas by a PostgreSQL advisory lock, so starting (or restarting) the console is all that's needed, both on first boot and on every release that ships new migrations. There is no separate migrate image or one-shot step to run.

To manage the schema out of band instead (e.g. in a regulated change window), set PGBC_AUTO_MIGRATE=false on the console and apply migrations yourself with goose / river against console-pg before starting it.

Verify the HTTP plane is up:

bash
curl -s https://pgbelaycrew.example.com/api/v1/healthz       # {"status":"ok"}
curl -s https://pgbelaycrew.example.com/api/v1/version
curl -s https://pgbelaycrew.example.com/api/v1/seal/status   # {"state":"uninitialized",...}

First-boot Init / Unseal

A fresh deployment starts with no master key in memory or on disk. Encrypted-data routes return 503 + X-Pgbc-Sealed: true until an admin completes the seal flow through the UI:

  1. Sign in via OIDC. Set PGBC_BOOTSTRAP_ADMIN_EMAIL first: the first user to sign in becomes global admin only if their email matches it and the IdP marked the address verified — otherwise they land as a viewer and nobody can complete the seal flow. Fail-closed when unset.
  2. The SealOverlay renders automatically. On the first boot it shows "Initialise master password" — choose a password (twice) and submit. The console generates a 32-byte master key, wraps it under an Argon2id-derived KEK, and stores the wrapped row as master_keys v1.
  3. On every subsequent restart the overlay shows "Console is sealed" instead — type the same password to unseal.

In both cases the CA, signing key, and agent gRPC plane lazy-load in the same process the moment Init/Unseal succeeds — no console restart needed.

No recovery

Save the password in your corporate password manager. There is no recovery: every wrapped blob (storage credentials, webhook HMACs, internal CA, signing keys) becomes unreadable if it is lost. See Master key (sealed) for the full lifecycle.

Option B — Kubernetes (Helm)

A starter chart ships at modules/backend/deploy/helm/console/. It is intentionally minimal — patch it for your platform (NetworkPolicy, ServiceMonitor, secret backend).

bash
helm install pgbelaycrew ./modules/backend/deploy/helm/console \
  --namespace pgbelaycrew --create-namespace \
  --set image.tag=<version> \
  --set existingSecret=pgbelaycrew-console \
  --set oidc.issuer=https://idp.example.com \
  --set oidc.clientId=pgbelaycrew-console \
  --set oidc.redirectUrl=https://pgbelaycrew.example.com/api/v1/auth/callback

Create the referenced secret before installing — it must contain DATABASE_URL, PGBC_SESSION_SECRET, PGBC_OIDC_CLIENT_SECRET. Add PGBC_BOOTSTRAP_ADMIN_EMAIL to your patched values. Do not include a PGBC_MASTER_KEY. The console exposes 8080 (HTTP), 9090 (gRPC — must be reachable from every PG host), and 9091 (Prometheus).

2 · UI install

The UI is a SvelteKit application served as a Node container. It speaks to the console over HTTP at the same origin: production deployments put both behind one reverse proxy so cookies, CSRF, and CSP work without cross-origin gymnastics.

docker-compose.yml
services:
  ui:
    image: ghcr.io/pgbelaycrew/ui:<tag>
    environment:
      PORT: 3000
      HOST: 0.0.0.0
    ports:
      - "127.0.0.1:3000:3000"

Reverse-proxy rules (HAProxy / nginx / Traefik / Ingress) — UI is the default backend, console wins for these prefixes:

Path prefixBackend
/api/, /api/v1/healthz, /pki/console:8080
everything elseui:3000

3 · Agent install

One agent per PostgreSQL host. Distributed as a native OS package (preferred) or a container. The native package is preferred because the agent needs read access to PGDATA, peer auth to PostgreSQL via the Unix socket, the ability to run wal-g against host paths, and systemd restart-on-crash — all awkward inside a container.

Option A — OS package (recommended)

FormatDistros
.debDebian, Ubuntu
.rpmRHEL, Fedora, CentOS Stream, Rocky, Alma, openSUSE
.apkAlpine
.pkg.tar.zstArch
.tar.gzanything else (manual install)

Each format ships in amd64 and arm64.

bash
# Debian / Ubuntu
sudo apt install ./pgbelaycrew-agent_<ver>_amd64.deb
# RHEL / Fedora / Rocky
sudo dnf install ./pgbelaycrew-agent-<ver>-1.x86_64.rpm
# Alpine
sudo apk add --allow-untrusted ./pgbelaycrew-agent_<ver>_amd64.apk
# Arch
sudo pacman -U ./pgbelaycrew-agent-<ver>-1-x86_64.pkg.tar.zst
# Generic tarball
sudo tar -C /usr/local/bin -xzf pgbelaycrew-agent_<ver>_linux_amd64.tar.gz

The post-install creates the pgbelaycrew system user (member of postgres), provisions /etc/pgbelaycrew, /var/lib/pgbelaycrew, /var/log/pgbelaycrew, copies agent.toml.example to agent.toml if no config exists, and registers the systemd unit.

Provision the observer role on every PostgreSQL instance the agent will manage:

bash
sudo -u postgres psql -f /usr/share/pgbelaycrew/observer-role.sql

This creates pgbelaycrew_observer — a least-privilege role for discovery and health checks. Edit /etc/pgbelaycrew/agent.toml, then enroll and start:

bash
# Mint a one-shot enrollment token in the console UI
# (Settings → Agents → Enroll a new agent), then on the PG host:
sudo -u pgbelaycrew pgbelaycrew-agent enroll \
    --token <one-shot-token> \
    --console https://pgbelaycrew.example.com

sudo systemctl enable --now pgbelaycrew-agent
sudo systemctl status pgbelaycrew-agent
journalctl -u pgbelaycrew-agent -f

The enrollment token is single-use. It carries the console URL and a fingerprint of the console CA, so the agent can validate the certificate it gets back. After enrollment the agent stores the issued client cert under /etc/pgbelaycrew/agent.crt (key under agent.key).

Option B — Container

Use this only if the agent's host requirements (PGDATA, sockets, wal-g binary) are already mounted into the container — typically only inside the validation harness.

bash
docker run -d \
  --name pgbelaycrew-agent \
  --network=host \
  -v /var/lib/postgresql:/var/lib/postgresql:ro \
  -v /var/run/postgresql:/var/run/postgresql \
  -v /etc/pgbelaycrew:/etc/pgbelaycrew \
  -v /var/lib/pgbelaycrew:/var/lib/pgbelaycrew \
  ghcr.io/pgbelaycrew/agent:<tag>

Verify the agent reached the console

In the UI, Settings → Agents lists the agent as online within ~10 seconds of systemctl start. If it stays unenrolled or offline, see Troubleshooting — the common causes are clock skew, CA mismatch, and firewalls blocking the gRPC port.

4 · Post-install checklist

Once console + UI + at least one agent are running:

  • You can sign into the UI via your real OIDC provider — not via Dex.
  • Settings → Agents shows every enrolled agent as online.
  • Settings → Storage backends has at least one S3 bucket declared and probed (Object Lock + versioning verified).
  • Each managed cluster has its agent attached and discovery reports green health.
  • curl https://pgbelaycrew.example.com/api/v1/healthz returns 200.
  • Prometheus is scraping /metrics on the console (:9091) and on every agent (127.0.0.1:9234).

02 · Configure guide

Configuring each component

Two surfaces: the console (environment variables, process restart to apply) and the agent (a TOML file at /etc/pgbelaycrew/agent.toml, hot-reloaded on SIGHUP).

A few things that look like configuration are deliberately not in either — they live in the UI, persisted in the console state DB, scoped per cluster:

  • Storage backends (S3 buckets, credentials, retention) — encrypted at rest under the sealed master key, rotatable without redeploy.
  • Backup schedules and verification cron — per cluster.
  • OIDC ↔ role mapping beyond client/secret — claims-to-role rules.
  • Compliance framework toggles.
  • Master key (sealed) — generated on first boot, wrapped under an operator-supplied unseal password. Not an env var.
By design

Configuration operators routinely change should not require a deploy. Configuration that affects the security boundary (the master-key seal, OIDC trust, mTLS PKI) does.

Console configuration

Mandatory in production

Set PGBC_ENV=production and the console refuses to start without these. Defaults that work in dev are dangerous in prod — that is intentional.

Console → managed-PG credentials

For paths where the console connects directly to a managed PostgreSQL host it builds a management DSN from PGBC_PG_SUPERUSER (default postgres), PGBC_PG_SUPERUSER_PASSWORD, and PGBC_PG_SSLMODE. In production the password must be non-default and sslmode=disable is refused.

OIDC discovery on a split network

When the issuer URL browsers see is not reachable from the console (split-horizon DNS, an IdP behind an internal hostname, Dex-in-Docker), point discovery at the in-network address and keep accepting tokens minted for the public issuer:

PGBC_OIDC_SPLIT_HOST_ALLOWLISTemptyRequired when discovery differs from the issuer. CSV of issuer=discoveryURL pairs allowed to be rewritten. Without a match the console refuses to start.

Master key (sealed)

The console no longer accepts a master key from env. On first boot the seal manager reports state=uninitialized; on every subsequent boot state=sealed, refusing encrypted-data routes (503 + X-Pgbc-Sealed: true) until the operator submits the unseal password.

StepWhenWhat happens
InitOnce, very first bootOperator chooses a password (twice). Backend generates a 32-byte master key, wraps it under an Argon2id-derived KEK, inserts master_keys v1. State → unsealed. CA + gRPC plane start.
UnsealAfter every restartOperator types the existing password. Backend unwraps every non-purged row into memory. State → unsealed.
RotateQuarterly / suspected leakA second admin approves. New key generated; old version stays decryptable until the background re-encryption walk completes, then retired.
RevokeKey compromiseSame shape as Rotate, but the old version is forensically purged after re-encryption — wrapped_key overwritten, status purged.
Operational consequences

A console restart leaves the system temporarily unusable until an admin unseals it — plan unattended restarts with your on-call rotation. There is no recovery if the password is lost. The unseal password lives only in operator memory + the running backend process; KMS-backed unsealing is not wired in this release.

UI configuration

The UI container is intentionally near-zero-config. Two environment variables matter:

VariableDefaultPurpose

Agent configuration

Single TOML file: /etc/pgbelaycrew/agent.toml, created from agent.toml.example by the package post-install. Hot-reloaded on SIGHUP.

Service user (auto-detected at install)

Condition at installRuntime userNotes
getent passwd postgres succeedspostgrespg_ctl, wal-g backup-fetch, and PGDATA writes run natively — no sudo shim.
otherwisepgbelaycrewDedicated low-privilege user (with postgres secondary); smaller blast radius.

[console] — how the agent reaches the control plane

agent.toml
[console]
url        = "https://pgbelaycrew.example.com"
cert_path  = "/etc/pgbelaycrew/agent.crt"     # written by `agent enroll`
key_path   = "/etc/pgbelaycrew/agent.key"
ca_path    = "/etc/pgbelaycrew/console-ca.pem"
heartbeat_interval     = "10s"
reconnect_max_backoff  = "60s"
subject_cn = "pgbelaycrew.example.com"

url, ca_path, and subject_cn are populated by pgbelaycrew-agent enroll — re-enroll rather than editing them by hand. heartbeat_interval should be ≤ half of the console's PGBC_HEARTBEAT_TTL.

[postgres] — how the agent reaches its local PG

agent.toml
[postgres]
socket_dir          = "/var/run/postgresql"
observer_role       = "pgbelaycrew_observer"   # provisioned by observer-role.sql
stats_interval      = "30s"
discovery_interval  = "5m"
discovery_cache_ttl = "4m"

Default is peer auth via Unix socket as the least-privilege observer role — no password, no DSN. The role can read configuration views, run pg_stat_* queries, and create restore points; it cannot read user data tables.

[walg], [storage], [sandbox]

agent.toml
[walg]
binary_path = "/usr/local/bin/wal-g"
config_file = "/etc/pgbelaycrew/walg.json"     # generated from the UI storage backend
[walg.env]
WALG_COMPRESSION_METHOD = "zstd"
WALG_UPLOAD_CONCURRENCY = "8"

[storage]
disk_temp_path          = "/var/lib/pgbelaycrew/tmp"
backup_space_multiplier = 4.0                  # refuses if <DB size> × N not free

[sandbox]
host_driver_enabled = false   # true → run as a postgres process instead of Docker
docker_socket       = "/var/run/docker.sock"
work_dir            = "/var/lib/pgbelaycrew/sandbox"
hard_ttl            = "4h"    # hard cap: older sandboxes are reaped
gc_interval         = "5m"

The agent does not bundle wal-g — install the version your team is on (check pgbelaycrew-agent diag). backup_space_multiplier is the safety factor applied before starting a backup.

[drift] — config-advisor enforcement

agent.toml
[drift]
mode = "report-only"   # report-only | pause-on-drift | enforce
  • report-only — log + report drift in the UI, take no action. Default.
  • pause-on-drift — pause scheduled backups for that cluster until drift is resolved or waivered.
  • enforce — re-apply the desired value via ALTER SYSTEM. Only on clusters where pgBelayCrew is the canonical source of truth.

Validate before starting

bash
sudo -u pgbelaycrew pgbelaycrew-agent config check            # parse + lint
sudo -u pgbelaycrew pgbelaycrew-agent config print --redact   # resolved values
sudo -u pgbelaycrew pgbelaycrew-agent diag                    # environment check
sudo -u pgbelaycrew pgbelaycrew-agent doctor                  # propose fixes

# After editing agent.toml, hot-reload without dropping the gRPC stream:
sudo systemctl reload pgbelaycrew-agent
03 · User guide

The operator's day

For DevOps / SRE operators running a fleet of PostgreSQL clusters. This guide is organized around the four primitives the UI is built on, plus the surrounding facilities (verifications, the advisor, audit, compliance).

How the UI is organized

AreaWhat you do here
ClustersThe map of every PostgreSQL deployment under management. Health, advisor findings, RPO/RTO, recent activity.
BackupsSchedules, on-demand runs, live progress, retention.
Restore points & runsAnchored states + the four restore intents (sandbox / recover-data / restore-cluster / emergency).
VerificationsL1 → L4 pipelines, evidence, attestations.
Compliance & audit33 controls × 5 frameworks, signed audit log, on-demand bundles.

Roles map to permissions

viewer

Read everything, change nothing.

operator

Declare clusters, run backups, take restore points, kick off sandbox / recover-data / drill restores.

admin

Manage agents, storage backends, OIDC mapping, retention, RBAC. Approves destructive restores.

auditor

Like viewer, plus exports compliance bundles and the signed audit log.

Separation of duty

Destructive actions (real takeover, emergency, cluster delete, backup delete, PKI/master-key rotation) all follow one path: a single second-person approval. The requester can never approve their own request, so a granted approval always represents two distinct humans. A human-readable reason lands in the audit log. There is no "force" flag.

1 · Cluster — registering and watching a PG deployment

A cluster is a PostgreSQL deployment (single node, or primary + standbys) discovered via an enrolled agent.

Enroll the agent first, then declare the cluster

  1. Settings → Agents → Enroll a new agent. Generates a single-use enrollment token. Copy it — it is shown once.
  2. On the PG host, pgbelaycrew-agent enroll --token <…>.
  3. Refresh Settings → Agents. The agent shows up as online.
  4. Clusters → Declare cluster. Pick the agent, give the cluster a name + a tag set, attach a storage backend.

Tag clusters with what matters operationally — environment (prod, staging), criticality (tier-1), data class (pii, phi). Schedules and verifications can target by tag.

What the cluster page tells you

  • Health — is the agent talking? Is PG up? Is the WAL archive flowing?
  • Advisor findings — 15 rules. Each has a severity, an impact statement, and the exact ALTER SYSTEM to fix it. Common early ones: archive_mode=off, full_page_writes=off, missing %p in archive_command, data_checksums=off, recent archiver failures.
  • RPO / RTO — measured numbers, not configured ones. RPO is the wall-clock distance between now and the last archived WAL; RTO is the time of the most recent successful restore drill.
  • Recent activity — backups, verifications, restore points, restore runs.
Before scheduling backups

Walk every advisor finding. Fix or waiver. A waiver is an explicit, signed acknowledgment that you accept the risk; it shows up in the audit log and on the compliance page until removed. Do not ship a waiver without writing the why.

2 · Backup — scheduling and running

A backup is a wal-g full or delta run against a cluster. The agent does the I/O; the console schedules and tracks.

Set the schedule

Cluster → Backups → Schedule. Pick:

  • Cadence — typically a daily full + 15-minute deltas. Cron syntax.
  • Retention — how many fulls to keep (and the deltas they anchor). Enforced server-side.
  • Window — optional time-of-day fence, useful when peak load is intolerant.

Retention is append-mostly: the policy decides what's deletable, but everything until then is immutable. With Object Lock enabled (recommended), even the console can't undo retention before its expiry.

On-demand backup

Cluster → Backups → Run now. Live progress streams over Server-Sent Events — bytes uploaded, files archived, ETA. If the console crashes mid-run, the job resumes: backups are durable river jobs, not detached goroutines.

What "succeeded" actually means

  • The wal-g run returned 0.
  • The console fetched the manifest and recorded real archive sizes (not what wal-g announced).

If either is missing, the backup is partial and counts as a failure for the SLO. A full backup additionally auto-enqueues an L2 verification when the cluster marks L2 as auto-after-full — tracked separately, and it does not retroactively flip the backup's own status.

3 · Restore point — naming a moment you can restore to

A restore point is a named anchor: an INSERT INTO pg_create_restore_point(name) recorded as a row, optionally with a fresh full backup and a verification run. Take it before a risky deploy, a schema migration, a contentious data fix.

ModeWhat it doesWhen to use
lightMarker onlyThe deploy is reversible by code; you just want a clean PITR target.
enoughMarker + delta backupThe deploy touches data; a delta is cheap insurance.
safeMarker + full backupSchema migration, ETL run — anything where "delta + WAL replay" is not a path you want under pressure.
paranoidMarker + full + L2 verificationPre-production cutover, irreversible migration. The verification proves the freshly-anchored backup actually restores.

Take it from Cluster → Restore points → New. Reason is mandatory and goes in the audit log. PostgreSQL gives you pg_create_restore_point(...); pgBelayCrew adds the operational shell around it — the marker is recorded server-side, surfaced in the UI, attached to the anchoring backup, and (in paranoid mode) tested before you walk away.

4 · Restore — the four intents

Restoring is the stressful path. The wizard collapses every option into one of four intents, named for what the operator actually wants.

IntentOutcomeApproval
sandboxStand up a side cluster (Docker or host driver), restore into it, leave it browsable. Source untouched.None — read-only side effect.
recover-dataSelective copy-back of specific tables/schemas from a sandbox into the source.None — reversible copy-back.
restore-clusterDrill (non-prod target) or real takeover (production).Single second-person approval for prod; none for drill.
emergencyFast takeover for true incidents.Single second-person approval. Logged loudly.

Sandbox — the answer 90% of the time

Your default for "I need to look at the data as of yesterday at 14:00":

  1. Cluster → Restore → Sandbox. Pick the target time (or a named restore point). The wizard shows the source backups it will use and the WAL range it will replay.
  2. Dry-run. The wizard prints the command sequence and the disk reservation. No commit yet.
  3. Confirm. A sandbox container (or host process) comes up restored. The UI gives you a connection string scoped to that sandbox.
  4. Browse, query, run consistency checks. When done, Reap — the sandbox is destroyed; the only artifact is the run record.

Sandboxes have a hard_ttl (default 4h) and are reaped automatically.

Upgrade test — restore into a newer major version

A sandbox restore with a target PG version set becomes an upgrade test: the wizard restores the backup, then loads it into a newer PostgreSQL major version and checks extension parity. Pick the method — dump-restore (default) or pg_upgrade (in-place). A successful upgrade test in the last 365 days satisfies the two upgrade-rehearsal compliance controls (SOC 2 CC8.1, ISO A.12.3.1). Map each major version's bin directory under [restore.upgrade.bin_dirs] in agent.toml.

Recover-data — copy a few tables back

  1. Take a sandbox to the moment-before. Confirm the data you want.
  2. Restore → Recover data. Pick tables / schemas. The wizard runs pg_dump from sandbox → pg_restore into source under a transaction.
  3. One operator approves. Audit log records exactly which rows came from where.

This is conservative: it is a copy, not a rewrite. Conflicts (PK collisions) are reported, not silently overwritten.

Restore-cluster — drill or real takeover

  • Drill (target = non-prod cluster) — runs end-to-end, measures real RTO, records the result in the cluster's drill history. Run this on a schedule. The first time someone restores should not be during the incident.
  • Real takeover (target = the production cluster itself) — promotes a restored sandbox into the production data path. Requires a single second-person approval. Refuses if anyone is mid-write to the source unless you explicitly fence first.

Emergency — the expedited path

Reserved for true incidents. Like every destructive action it needs one approval from a second person — separation of duty is never bypassed — but it skips the review window and other ceremony. The audit log records who, when, what was bypassed, and what reason was given, and the whole org is notified by webhook + email. If you use this more than once a quarter, the policy needs revisiting.

5 · Verifications — proving backups actually restore

Backups that are never restored are a liability. The verification ladder runs on a schedule (or auto-triggered after every full), produces a signed attestation, and emits the result.

Release status

In this release the verification pipeline ships in simulation mode by default. A run does not fabricate a pass or sign an attestation — it reaches an honest simulated terminal state (with the live → terminal + RTO flow intact) until the real agent-driven pipeline is enabled (PGBC_SIMULATE_VERIFY, off in production). Treat the levels below as the target behavior.

LevelWhat it doesCostFrequency
L1basebackup checksumsecondsEvery backup, inline
L2full sandbox restore + assertionsminutesDaily / per-full
L3regression suite against the restored sandboxminutes–hoursWeekly
L4multi-cluster (cross-site replay)hoursMonthly / quarterly

Configure per cluster, per level in Cluster → Verifications. Two mutually-exclusive modes:

  • Cron — fixed cadence. Predictable, decoupled from backup timing.
  • Auto-after-full — runs immediately after every scheduled full backup. Higher signal, costs more compute.

Each run produces a pass/fail with an evidence pointer, a signed attestation (cosign signature over the run record), and an audit-log entry. A failed verification does not automatically pause backups — it raises an alert. Combine with [drift].mode = pause-on-drift for autonomous safety.

6 · Configuration advisor — fix what wal-g can't

The advisor evaluates 15 rules against the live PG configuration. Each finding includes a severity (critical / warning / info), an impact, the exact fix (ALTER SYSTEM, an initdb flag, or a restart note), and a citation from the PostgreSQL or wal-g docs.

You can apply a fix in two clicks (the advisor runs the SQL via the agent, dual-approving if superuser is implied), dry-run it, or waiver it. Drift detection re-runs the advisor every few minutes; if a value changes outside the advisor's flow, the cluster shows a drift badge until you re-apply, accept, or waiver. Behavior is governed by [drift].mode in agent.toml.

7 · Audit log

Every action — login, backup trigger, restore approval, configuration change, waiver, agent enrollment, role change — appends to a hash-chained log. Each entry contains the previous entry's hash; tampering with any row breaks the chain and the next verification fails.

  • Browse and filter — by user, action, cluster, time range.
  • Verify the chain on demand — produces a pass/fail with the bad row, if any.
  • Export a signed bundle — the audit chain + attestations + storage probes + control evaluations. This is what auditors download.
No DELETE

There is no DELETE on the audit log. State changes; rows don't disappear. Same applies to attestations, configuration history, restore runs, and verifications. If you ever need to "purge" something for legal reasons, that's a topic for the on-call legal contact, not a button in the UI.

8 · Compliance — controls evaluated by real queries

33 controls across SOC 2, ISO/IEC 27001, HIPAA, GDPR, PCI DSS. Every one is a real SQL query against state pgBelayCrew owns — no mocked statuses, at most one DB query stale.

  • "Backups exist and are recent" → backups table, last_success per cluster.
  • "Restores have been tested" → drill history, last_drill within the policy window.
  • "Storage is immutable" → Object Lock probe on the bucket registry.
  • "Audit log is intact" → live hash-chain verification.
  • "Privileged actions require approval" → restore_runs joined with approvals.
  • "Backups load into a newer PG major version" → the two upgrade-rehearsal controls pass when a successful upgrade-test restore ran in the last 365 days.

The compliance page is a per-cluster matrix — a control can pass for cluster A and fail for cluster B. Filter by framework, by cluster tag, by status. For findings you can't fix immediately, the gap register records a waiver with an owner and a deadline; waivers expire and re-open as findings. The 90-day trend is the chart auditors actually look at.

Producing a bundle for an auditor

Compliance → Export bundle. Pick the frameworks, clusters, time range. The console produces a signed .zip containing the control matrix at export time, the evidence rows behind each control, the signed audit-log slice for the period, and a signature over the whole bundle. Auditors verify the signature; the bundle stands on its own.

9 · Day-to-day rhythm

A reasonable weekly operator cadence, once the fleet is set up:

CadenceAction
DailySkim the Clusters dashboard, triage any red. Check backup success for the prior 24h.
WeeklyWalk advisor findings — clear or waiver each. Review the verification results.
WeeklyRun a sandbox restore on one randomly-picked cluster. Real RTO; real confidence.
MonthlyDrill a full restore-cluster on staging. Note the RTO; chart it.
QuarterlyExport a compliance bundle, even if no audit is pending. Forces you to look at gaps.

What you should not be doing weekly: restarting agents (they hot-reload), hand-editing agent.toml on each host (declare once, push via config management), or computing RPO/RTO manually for the audit (the console computes them; the export contains them).

04 · Troubleshooting

1 · Agent stays unenrolled or offline

Operator-facing failure modes, organized by symptom — start at the section that matches what you're seeing. Many "broken" symptoms are really first-install mistakes; if you're still bringing the system up, re-check Install and Configure first.

Step 1 — confirm what the agent thinks

PG host
sudo systemctl status pgbelaycrew-agent
sudo journalctl -u pgbelaycrew-agent -n 200 --no-pager
sudo -u pgbelaycrew pgbelaycrew-agent diag
sudo -u pgbelaycrew pgbelaycrew-agent config print --redact

Read the journal before restarting anything — restarts hide the original error.

Step 2 — match the symptom

Journal symptomCauseFix
enrollment token expired / token already usedOne-shot tokens. Ran enroll twice, or waited too long.Mint a fresh token (Settings → Agents → Enroll), re-run enroll.
x509: certificate signed by unknown authorityThe agent does not trust the console's server cert.Re-fetch the console CA. Token enrollment carries the CA fingerprint; for a manual console-ca.pem, replace it with what the console presents.
x509: certificate has expired or is not yet validClock skew.Fix NTP. ±5 minutes is the practical tolerance.
connection refused / i/o timeout to gRPCThe console's :9090 is not reachable from this host.Open the firewall. Don't put a layer-7 proxy in front; gRPC streams are long-lived.

Step 3 — when nothing matches

Cleanly re-enroll. Non-destructive — the agent's local state DB is preserved:

PG host
sudo systemctl stop pgbelaycrew-agent
sudo -u pgbelaycrew pgbelaycrew-agent unenroll --keep-logs
# Mint a new one-shot token in the UI.
sudo -u pgbelaycrew pgbelaycrew-agent enroll --token <new> --console https://pgbelaycrew.example.com
sudo systemctl start pgbelaycrew-agent

2 · Agent cert revoked or expired

If revoked from the console

The agent's cert is on the revocation list; the console refuses the next handshake. The agent keeps retrying with backoff. Procedure:

PG host
sudo systemctl stop pgbelaycrew-agent
sudo rm /etc/pgbelaycrew/agent.crt /etc/pgbelaycrew/agent.key
# Mint a fresh token in the UI for this agent.
sudo -u pgbelaycrew pgbelaycrew-agent enroll --token <new> --console https://pgbelaycrew.example.com
sudo systemctl start pgbelaycrew-agent

The cluster the agent was attached to stays attached — cluster identity is server-side, not bound to the cert. The audit log records "agent re-enrolled".

Console server cert expired

The console mints its own serving cert from the internal CA and re-mints it at 2/3 of lifetime, so this should not happen. If it does, the renewal loop has likely been silenced by a recurring CA error — resolve the underlying issue and the next tick issues a new cert. Forced workaround: restart the console process to mint a fresh cert at startup.

3 · Backups failing or partial

"succeeded but partial"

The wal-g run returned 0 but a post-condition failed (manifest fetch, real-size accounting, or inline L1 verification). Look at the wal-g log:

PG host
ls -la /var/log/pgbelaycrew/walg/
sudo -u pgbelaycrew tail -200 /var/log/pgbelaycrew/walg/<latest>.log
  • Inline L1 checksum mismatch — corruption in the freshly-uploaded basebackup. Re-run; if it persists, suspect storage or RAM.
  • Manifest fetch failed — the bucket allowed PUT but is denying GET. Check IAM policy.

"failed: insufficient disk space"

[storage].backup_space_multiplier (default 4.0) refused to start because <DB size> × 4 is not free on disk_temp_path. Free space (the right answer), or lower the multiplier only if you know your compression actually achieves it.

"archive_command failed"

PG host
sudo -u postgres psql -c "SELECT * FROM pg_stat_archiver;"

failed_count > 0 with a recent last_failed_time — most often bucket credentials rotated without refreshing the wal-g config (re-declare the storage backend in the UI), or the pgbelaycrew user lost group access to PGDATA.

4 · Restore won't proceed

"approval required, but I am the only operator"

Production restore-cluster, emergency, and cluster-delete each require one approval from a second person — the requester can never approve their own request. There is no force flag. Approve as a second user (the right answer for any path); emergency still needs that approval but skips the review window and logs loudly. If you routinely hit this in non-emergency, your org has a process problem (single SRE on call), not a tool problem.

"sandbox failed to start"

  • Docker driver — docker_socket not accessible to the agent user. The post-install adds pgbelaycrew to postgres, not docker. Either gpasswd -a pgbelaycrew docker or switch to the host driver.
  • Host driver — needs a postgres binary on PATH and writable [sandbox].work_dir. diag reports both.

"sandbox reaped while I was using it"

[sandbox].hard_ttl (default 4h) is a hard cap. Raise it in agent.toml before you start the sandbox; reloading mid-sandbox doesn't extend the existing TTL.

5 · Console DB is down

The console fails to start, or returns 500 with database connection failed.

If the console is misconfigured

console env
echo "$DATABASE_URL"                  # right host? right sslmode?
psql "$DATABASE_URL" -c "select 1;"   # does it actually connect?

Most outages are configuration: rotated DB password, expired DB cert in verify-full, or a pgBouncer that doesn't support the wire features the console uses. Don't point the console at a pgBouncer transaction pool — it uses LISTEN/NOTIFY and prepared statements that need a session pool.

If the DB is genuinely down

  • Agents keep working. Local SQLite state continues cron-driven backups. You lose live progress; the agent reports them when the console returns.
  • In-flight restore jobs (durable river jobs) resume when the console returns.
  • The audit chain is paused — it doesn't fork or break, it just doesn't grow.

If the DB host is permanently lost, restore a recent dump (the state DB is not backed up by pgBelayCrew — back it up like your other Postgres), point DATABASE_URL at the new host, start the console. Agents reconnect, cluster identities are preserved, the audit chain continues.

6 · Master key — sealed lifecycle

Console boots stuck on 503 + X-Pgbc-Sealed: true

The system is sealed. Sign in, the SealOverlay paints automatically, type the password. The CA + agent gRPC plane come up the moment unseal succeeds — no restart needed. If the overlay never appears, check /api/v1/seal/status; if it reports state=sealed but you don't see the modal, hard-refresh the tab.

Forgotten unseal password

There is no recovery. The Argon2id KEK is a one-way derivation; every wrapped blob is unreachable without the password, and restoring the DB doesn't help. If genuinely lost: wipe master_keys (or restore a pre-encryption dump), re-init with a new password, re-declare every storage backend, and re-enroll every agent (the internal CA private key was lost with the master key).

Suspected compromise (Revoke) vs routine rotation

Use Revoke when the password and the DB dump leaked: rotate every storage credential at the provider first, pull the new credentials in, then Settings → Security → Revoke (a second admin approves), supply the new master password, and the background job re-encrypts every blob under a fresh key and forensically purges the old version. Rotate is the same flow at lower urgency (previous version retired, not purged) — quarterly is reasonable, and the migration is online.

7 · wal-g version mismatch

diag warns wal-g version not in supported matrix, or backups fail on flags wal-g doesn't recognize, or old archives can't be read. The agent does not bundle wal-g — your distribution does.

  • Older than the matrix lower bound — backups may not be restorable by a current agent, and missing CLI flags break new features. Upgrade.
  • Newer than the upper bound — usually fine for backup, but new archive formats may not be readable downstream. Check release notes.
PG host
# Pause the schedule (UI) or wait for an idle window.
sudo systemctl stop pgbelaycrew-agent
# Install the new wal-g at [walg].binary_path (default /usr/local/bin/wal-g).
sudo systemctl start pgbelaycrew-agent
sudo -u pgbelaycrew pgbelaycrew-agent diag    # confirm matrix-OK

A failed in-place upgrade does not corrupt existing archives — the archive format is wal-g's, not the agent's.

8 · Audit log shows chain broken

Every few minutes the console verifies the hash-chained audit log. chain broken is a security event, not a bug.

bash
curl -s -X POST https://pgbelaycrew.example.com/api/v1/audit/verify | jq

The response includes the first bad row by ID. From there:

  1. Check who/what last wrote near that row (legit operator action, or an unexpected actor?).
  2. Check for direct writes to audit_log outside the console's API. The only legitimate writer is the console process; a DBA running an UPDATE for "cleanup" will break the chain.
  3. The chain cannot be "repaired" — that's the point. Record an addendum (a new entry with a chain_repaired_after reference and explanation), but the original break stays visible.

If the break has no explanation, escalate per your incident process. Bring the signed-bundle export from before the break.

05 · Build guide

Producing the artifacts you install

For whoever builds the release: agent OS packages and container images for console / UI / agent. Both are driven from the project-root Makefile. If you only want to install a release someone else built, go to Install instead.

Why two artifact types

  • OS packages install the agent on the PostgreSQL host itself, with a real systemd unit, a real pgbelaycrew user in the postgres group, peer auth via Unix socket, and direct filesystem reach to PGDATA. Five formats so it installs cleanly on every Linux distro.
  • Container image of the agent exists primarily for the validation harness — same code, packaged for a Docker network.

The console and UI are container-only. Everything builds for amd64 and arm64 — multi-arch is not optional.

Versioning

Every module's VERSION file holds the same string, in the form YY.MM.<short-sha>:

bash
$ make version
VERSION=26.05.2ce6d86
COMMIT=2ce6d86
BUILD_DATE=2026-05-08T19:10:00Z
REGISTRY=ghcr.io/pgbelaycrew
TAG=26.05.2ce6d86
PLATFORMS=linux/amd64,linux/arm64

The three modules are kept in lockstep — the gRPC contract evolves together. make bump-version writes the same YY.MM.<sha> into every module before tagging a release.

Agent packages — what make packages produces

Five formats × two arches = 10 artifacts in modules/agent/dist/.

Container images — what make images produces

Three images, each with two tags (<TAG> and latest).

TargetImage

  • console — golang:1.25-alpine → distroless. EXPOSE 8080 9090 9091.
  • ui — node:20-alpine runs npm run build:prod (strict mock gate) → node:20-alpine runtime. Runs as node, not root.
  • agent — builder → ubuntu:24.04 (not distroless: bundled wal-g is built against glibc). Bundles wal-g; EXPOSE 9234.

The schema (goose) and queue (river) migrations are embedded in the console image (migrations/embed.go) and applied in-process at startup, so there is no separate migrate image to build or publish.

Single-arch (host) vs multi-arch (push)

bash

These call into per-module Makefiles and drive the same logic CI uses — a green local run means a green CI run.

Reproducibility — what is and isn't pinned

  • Go toolchain — pinned to go.mod (golang:1.25-alpine for images, golang:1.24-bookworm for package cross-compilation).
  • Node — pinned to node:20-alpine.
  • nfpm — goreleaser/nfpm:latest (pin for byte-identical builds).
  • Runtime bases — console distroless; agent ubuntu:24.04. All tags roll; pin to a digest for supply-chain reproducibility.
  • wal-g (agent image) — --build-arg WALG_VERSION=latest resolves at build time. Pin WALG_VERSION (and optionally WALG_SHA256) explicitly.
  • COMMIT — embedded from git rev-parse --short HEAD. Build inside a non-git checkout and you get unknown. Don't.
Common build failures

buildx not available → install the docker-buildx plugin. Multi-arch image won't load locally → expected; buildx build --push can't load a multi-arch manifest, build single-arch with make image-*. UI build fails with "mock gate" → ensure VITE_USE_MOCK and VITE_MOCK_AUTH are unset/0. Registry push 401/403 → docker login first.
